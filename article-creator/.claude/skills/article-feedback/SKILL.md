# 記事フィードバック収集・蓄積スキル

PRレビューでの指摘事項を分析し、次回以降の記事作成に自動反映される永続ルールとして `.claude/rules/article-feedback.md` に保存する。

## 動作フロー

### ステップ1: フィードバック収集

以下のいずれかからフィードバックを収集する：

1. **PRレビューコメント** — `gh api` でPRのレビューコメントを取得
2. **issueコメント** — ユーザーが直接指摘した内容
3. **ユーザーの直接指示** — 「こういう点を改善して」など

```bash
# PRのレビューコメントを取得
gh api repos/{owner}/{repo}/pulls/{pr_number}/comments --jq '.[].body'
# PRのレビュー（approve/request changes含む）を取得
gh api repos/{owner}/{repo}/pulls/{pr_number}/reviews --jq '.[] | select(.body != "") | .body'
# issueのコメントを取得
gh api repos/{owner}/{repo}/issues/{issue_number}/comments --jq '.[].body'
```

### ステップ2: フィードバック分析

収集したコメントから **記事品質に関するフィードバック** のみを抽出する。

以下のカテゴリに分類：

| カテゴリ | 例 |
|---------|------|
| 文体・トーン | 「もっとカジュアルに」「硬すぎる」 |
| 構成・流れ | 「導入が長い」「結論が弱い」 |
| 情報の質 | 「具体例が少ない」「数値を入れて」 |
| 読みやすさ | 「段落が長い」「箇条書きを増やして」 |
| SEO・タイトル | 「タイトルが抽象的」「キーワードが不自然」 |
| 画像・ビジュアル | 「画像が少ない」「アイキャッチが合ってない」 |
| ファクト | 「情報が古い」「出典を明記して」 |
| フォーマット | 「note形式に合ってない」 |

コードレビューやワークフローに関するコメントは除外する。

### ステップ3: ルール保存

抽出したフィードバックを **抽象化** してから `.claude/rules/article-feedback.md` に追記する。

#### 抽象化のプロセス

ユーザーの具体的な指摘を、あらゆる記事に適用できる汎用ルールに昇華する。

**例:**
| ユーザーの指摘（具体） | 保存するルール（抽象） |
|---|---|
| 「この記事の冒頭が退屈」 | 導入は読者の課題や疑問から始め、3行以内で引き込む |
| 「NISAの説明が雑」 | 制度・サービス名を出す場合は読者が行動できるレベルまで具体的に説明する |
| 「この段落が長すぎて読む気にならない」 | 1段落は3〜4文（100文字前後）を上限とし、それを超える場合は分割する |
| 「数字の根拠がない」 | 統計・数値を記載する場合は必ず出典または時点を併記する |
| 「タイトルがつまらない」 | タイトルに「数字」「疑問形」「読者メリット」のいずれかを含める |

**抽象化の原則:**
- 「この記事の○○」→「すべての記事で○○」に変換する
- 特定のトピック名を消し、パターンとして記述する
- 「〜しないで」（禁止）より「〜する」（推奨）の形で書く
- 判断基準が曖昧にならないよう、数値や条件を含める

**ルールの書式:**

```markdown
### [カテゴリ] ルール内容（YYYY-MM-DD）
- 元の指摘: ユーザーの原文（どんな文脈で出た指摘か記録）
- ルール: 抽象化した汎用ルール
- 適用先: どのエージェント（執筆/校正/構成/デザイン等）が意識すべきか
```

**重要なルール:**
- 既存のルールと重複する場合は追加せず、既存ルールを更新する
- 矛盾するフィードバックがある場合は新しい方を優先し、古い方を削除する
- 特定の記事にしか当てはまらない指摘は **抽象化できる場合のみ** ルール化する
- 抽象化できない一回限りの指摘はルールにしない

### ステップ4: 確認

追加・更新したルールをユーザーに報告する。

```
## フィードバック記録結果

追加したルール:
- [文体] 専門用語を使う場合は必ず簡単な説明を併記する
  ← 元の指摘: 「NISAの説明が初心者にはわかりにくい」を抽象化

更新したルール:
- [構成] 導入は3行以内にする → 導入で読者の課題提示+3行以内で引き込む（具体化）
  ← 元の指摘: 「この記事の冒頭が退屈」を抽象化

スキップしたコメント:
- 「CIが通ってない」→ 記事品質に関するフィードバックではないためスキップ
- 「2段落目のリンクが切れてる」→ この記事固有の問題であり、抽象化できないためスキップ
```

## 入力

- PRのURL or PR番号（PRレビューから収集する場合）
- または直接のフィードバックテキスト

## 出力

`.claude/rules/article-feedback.md` への追記・更新

## 注意事項

- このファイルは `.claude/rules/` に配置されているため、Claude Codeの全セッションで自動的に読み込まれる
- blog-writer-multi-agent の全エージェント（特に執筆・校正・構成・デザイン担当）がこのルールを参照する
- ルールが増えすぎた場合（20件超）は、古いものや重要度の低いものを統合・削除して整理する
