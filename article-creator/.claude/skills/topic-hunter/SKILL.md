# トピックハンター — マルチエージェント記事ネタ発掘スキル

複数のエージェントが並列でWeb検索とX（Twitter）バズ検索を行い、3カテゴリ（プログラミング・健康・資産形成）から差別化された記事ネタを発掘する。

## 過去記事との重複管理（必須）

ネタ探索の **前** に `published_articles.tsv` を読み込み、過去に投稿済みの記事を把握した上でネタを選定すること。

### TSVフォーマット（1行1記事、タブ区切り）
```
カテゴリ	日付	slug	タイトル
```

### 手順
1. `published_articles.tsv`（プロジェクトルート直下）を読む（数行のテキスト）
2. 探索エージェント（Tech/Health/Asset）に、該当カテゴリの行だけ渡す
3. カテゴリ内深掘りアナリストには全行を渡す
4. ネタ提案時に、過去記事との関係性（完全新規 / 深掘り / 続編）を明記する

### 判断基準（3段階）

#### 1. 完全除外（NG）
- **タイトル・主題がほぼ同じで、新しい情報や切り口がない** もの
  - 例: 過去に「iDeCo改正2026」→ 同じ内容を繰り返すだけの「iDeCo改正まとめ」はNG

#### 2. 深掘り・続編として許可（OK）
- 過去記事と同じテーマでも、**追加の情報・新しい視点・最新の動向** を加えられるなら許可
- この場合、記事内で **過去記事へのリンクを掲載** し、読者が前提知識を得られるようにする
  - 例: 過去に「MCPとは？」→ 「MCP実践編: ○○を作ってみた」はOK（前回記事リンク付き）
  - 例: 過去に「iDeCo改正2026」→ 「iDeCo改正後の最適ポートフォリオ」はOK
- ネタ提案時に `related_past_article` フィールドで過去記事の slug を指定すること

#### 3. 完全新規（最優先）
- 過去記事と関連しないテーマは最優先で採用する

### 探索エージェントへの追加指示テンプレート
各探索エージェント（Tech/Health/Asset）のプロンプト末尾に以下を追加する:
```
## 過去に投稿済みの記事（内容の重複に注意すること）
{published_articles.tsv の該当カテゴリの記事リスト}

- まったく同じ内容の繰り返しは除外する
- 過去記事を深掘り・発展させるネタは歓迎（その場合 related_past_article を指定）
- 完全新規のテーマを優先する
```

### ネタ提案フォーマット（追加フィールド）
```
### 1. [タイトル案]（カテゴリ: tech）
コンセプト: ...
差別化ポイント: ...
推定関心度: ★★★
過去記事との関係: 完全新規 / 深掘り（slug: xxx）
```

## SocialData API（X/Twitter バズ検索）

各探索エージェントは、Web検索に加えて `./scripts/socialdata_search.sh` を使い、
X上でバズっている投稿からもトレンドを収集する。

```bash
# 使い方
./scripts/socialdata_search.sh "検索クエリ" [type] [count]
# type: "Top"（バズ投稿）or "Latest"（新着）、デフォルト Latest
# count: 取得件数上限、デフォルト 20

# 例: AI関連のバズ投稿を5件取得
./scripts/socialdata_search.sh "AI agent" Top 5

# 例: いいね100以上のプログラミング投稿
./scripts/socialdata_search.sh "#プログラミング min_faves:100" Top 10

# 例: 日本語のNISA関連バズ投稿
./scripts/socialdata_search.sh "NISA lang:ja min_faves:50" Top 10
```

**環境変数**: `SOCIALDATA_API_KEY`（未設定時はスキップされる）
**コスト**: 約$0.004/回（20件取得時）
**各エージェントのAPI呼び出し**: 最大3回まで（コスト管理）

## 実行フロー

### フェーズ1: 並列ネタ探索（3エージェント同時）

3つのエージェントを **並列** で起動し、各カテゴリの最新トレンド・話題を収集する。

#### Agent 1: Tech探索エージェント

```
あなたはプログラミング・テクノロジー分野のトレンドリサーチャーです。
Web検索を使って、今話題の技術トピックを幅広く調査してください。

## 調査する情報源（複数を横断すること）
- 技術ブログ・ニュースサイト（Hacker News, TechCrunch, Publickey, Zenn トレンド等）
- GitHub Trending（注目リポジトリ・言語）
- 新しいフレームワーク・ツール・ライブラリのリリース情報
- AI/ML関連の最新動向
- セキュリティ関連のニュース
- 開発者コミュニティで議論されているテーマ
- **X（Twitter）でバズっている技術投稿**（SocialData API経由）

## X バズ検索（必ず実行すること）
以下のスクリプトでX上のバズ投稿を調査する（最大3回まで）:

**検索クエリは固定ではなく、動的に生成すること:**
1. まずWeb検索でTech分野の最新トレンドキーワードを把握する
2. 把握したキーワードを元に、2〜3個の検索クエリを自分で考える

5. 日本語クエリには `lang:ja` を付ける

**例（参考。実際は自分でトレンドに合ったクエリを考えること）:**
- "Claude Code min_faves:500"
- "Rust OR Go min_faves:200 lang:ja"
- "LLM agent framework min_faves:300"
バズ投稿（いいね・RT・表示数が多いもの）は audience_appeal を高く評価し、
sources に元ツイートのURLを含めること。

## 出力形式（JSON）
{
  "category": "tech",
  "trends": [
    {
      "topic": "トピック名",
      "summary": "概要（2-3文）",
      "why_interesting": "なぜ今面白いか",
      "sources": ["情報源URL or 名前"],
      "freshness": "today/this_week/this_month",
      "audience_appeal": "high/medium/low"
    }
  ],
  "emerging_patterns": ["新興パターンや横断的テーマ"],
  "search_date": "検索日"
}

最低8個のトレンドを見つけること。鮮度と多様性を重視する。
```

**入力**: なし（Web検索で自律的に調査）
**出力**: `{作業ディレクトリ}/_tech_trends.json`
**Web検索**: 必要

### フェーズ2: カテゴリ内深掘り分析（1エージェント）

各カテゴリの調査結果を分析し、同じカテゴリ内のトレンド同士を組み合わせて深みのある記事ネタを提案する。

**重要: 異なるカテゴリの情報を混ぜてはいけない。Tech の記事に Asset の話を混ぜたり、その逆も禁止。**

#### Agent 4: カテゴリ内深掘りアナリスト

```
あなたは記事ネタの深掘り分析の専門家です。
3つのカテゴリ（プログラミング、健康、資産形成）の調査結果をそれぞれ読み込み、
同じカテゴリ内のトレンド同士を組み合わせた、深みのある記事ネタを提案してください。

## 重要なルール
- **カテゴリを跨いだ組み合わせは絶対に禁止**
  - ❌ Tech × Asset（例: 「Pythonで家計管理」）
  - ❌ Tech × Health（例: 「AIで睡眠最適化」）
  - ❌ Health × Asset（例: 「医療費と資産形成」）
- **同じカテゴリ内のトレンドを組み合わせる**
  - ✅ Tech内: 「MCPの普及 × AIエージェント開発の民主化」→ 1本のTech記事
  - ✅ Health内: 「腸内環境の最新研究 × 睡眠の質向上」→ 1本のHealth記事
  - ✅ Asset内: 「iDeCo改正 × 退職所得控除の変更」→ 1本のAsset記事
- 1つのカテゴリに集中した、読者が深く理解できる記事にする

## 分析の方針
- 各カテゴリ内で、関連するトレンドをグルーピングする
- 単独では弱いトレンドも、同カテゴリの別トレンドと組み合わせると面白くなる場合がある
- 「このトレンドを知った読者が次に知りたいこと」を同カテゴリ内から探す

## 出力形式（JSON）
{
  "tech_ideas": [
    {
      "title_idea": "記事タイトル案",
      "concept": "コンセプト説明（3-4文）",
      "unique_angle": "他のブログとの差別化ポイント",
      "source_trends": ["組み合わせたトレンド名（すべてTech内）"],
      "estimated_interest": "high/medium/low",
      "difficulty": "easy/medium/hard"
    }
  ],
  "health_ideas": [
    {
      "title_idea": "記事タイトル案",
      "concept": "コンセプト説明（3-4文）",
      "unique_angle": "差別化ポイント",
      "source_trends": ["組み合わせたトレンド名（すべてHealth内）"],
      "estimated_interest": "high/medium/low",
      "difficulty": "easy/medium/hard"
    }
  ],
  "asset_ideas": [
    {
      "title_idea": "記事タイトル案",
      "concept": "コンセプト説明（3-4文）",
      "unique_angle": "差別化ポイント",
      "source_trends": ["組み合わせたトレンド名（すべてAsset内）"],
      "estimated_interest": "high/medium/low",
      "difficulty": "easy/medium/hard"
    }
  ]
}

各カテゴリから最低3個のアイデアを提案すること。
```

**入力**: `{作業ディレクトリ}/_tech_trends.json` + `_health_trends.json` + `_asset_trends.json`
**出力**: `{作業ディレクトリ}/_analysis.json`
**Web検索**: 不要

### フェーズ3: ネタ提案・選定（リード）

リードが `_analysis.json` を読み込み、以下を行う：

1. 各カテゴリのアイデアを推定関心度・鮮度・差別化度でランキング
2. **トップ5のネタ候補** をユーザーに提案（各候補は必ず1つのカテゴリに属する）
3. 各候補に「なぜこのネタが面白いか」の理由を付ける

提案フォーマット：
```
## 今日のブログネタ候補

### 1. [タイトル案]（カテゴリ: tech）
コンセプト: ...
差別化ポイント: ...
推定関心度: ★★★

### 2. ...
```

### フェーズ4: 記事作成・PR作成

ユーザーがネタを選択したら（または自動実行の場合は最も推定関心度の高いものを選択）：

1. 選択されたネタの情報を `test_config.json` 形式に変換
2. **blog-writer-multi-agent スキル** を呼び出して記事を作成
3. 記事作成完了後、以下を実行：
   - 新しいブランチを作成: `article/YYYYMMDD-slug`
   - 作成された記事（`outputs/` 配下）をコミット
   - リモートにプッシュ
   - `gh pr create` でPRを作成（記事の概要・ファクトチェック結果をPR本文に含める）
   - **PRの本文末尾に `Closes #<issue番号>` を必ず含めること。** ネタ候補issueがある場合、PRマージ時にissueが自動クローズされる

## 出力ディレクトリ

`outputs/YYYYMMDD_タイトル/` 配下に中間ファイルを出力（blog-writer-multi-agentと同じ構造）

### meta.json（デプロイ用メタデータ）

記事作成完了後、`outputs/YYYYMMDD_タイトル/meta.json` を生成すること。
deploy-article ワークフローがこのファイルを参照して writing-project にデプロイする。

```json
{
  "category": "tech",
  "slug": "ai-agent-trends-2026",
  "tags": ["AI", "エージェント", "トレンド"],
  "thumbnail": "https://images.unsplash.com/photo-xxx",
  "publish_approved": false,
  "published": false
}
```

- `category`: "tech" | "health" | "asset" のいずれか
- `slug`: URL用のスラッグ（英数字とハイフン）
- `tags`: 記事のタグ配列
- `thumbnail`: アイキャッチ画像URL（Unsplash連携で取得、なければ省略）
- `publish_approved`: ユーザーがPRを承認したらtrueになる
- `published`: デプロイ済みならtrue

## 自動実行モード（GitHub Actions用）

GitHub Actions から呼ばれる場合は、以下の動作をする：

1. フェーズ1-2を実行してネタを発掘
2. トップ候補のネタでGitHub issueを作成（ネタ一覧をissue本文に記載）
3. issueのラベル: `blog-topic`, カテゴリ名（`tech`, `health`, `asset`）
4. issue本文に全候補を記載し、ユーザーがコメントでネタを選択できるようにする

## 対話実行モード（CLI用）

CLIで直接呼ばれた場合は：

1. フェーズ1-2を実行
2. ネタ候補をユーザーに提示
3. ユーザーの選択を待つ
4. 選択後、blog-writer-multi-agent → PR作成まで実行
