# ナレッジコレクター

ユーザー（pao.cho）が自分自身について話している内容を分析し、適切なナレッジファイルに情報を追加する。
会話の中でユーザーの人となりが見える発言があったら、投稿の独自性・リアリティを高める素材としてナレッジに蓄積する。

## 起動条件

以下のいずれかに該当する発言をユーザーがしたとき、`/knowledge-collector` を実行する。

### 自動検知すべき発言パターン

| パターン | 例 | 蓄積先 |
|---------|-----|--------|
| **自分語り・自己紹介** | 「僕は〜なタイプ」「昔から〜」 | personal.json |
| **振り返り・内省** | 「今思うと〜」「あの時〜だった」「反省してるのは〜」 | personal.json, episodes.json |
| **思考・考えの共有** | 「僕は〜だと思う」「〜って大事だよね」 | opinions.json |
| **意見・スタンス表明** | 「〜には反対」「〜派です」「〜は間違ってると思う」 | opinions.json |
| **経験談・エピソード** | 「〜したことがある」「前に〜で」「〜のとき〜だった」 | episodes.json |
| **価値観・信念** | 「〜を大事にしてる」「〜が信条」「〜は譲れない」 | personal.json |
| **好み・趣味・習慣** | 「〜が好き」「毎朝〜してる」「最近〜にハマってる」 | personal.json |
| **スキル・専門知識の開示** | 「〜年やってる」「〜が得意」「〜を使ってる」 | domain.json, profile.json |
| **目標・将来の展望** | 「〜したい」「将来は〜」「次は〜に挑戦」 | personal.json |
| **失敗・学びの共有** | 「失敗したのは〜」「〜から学んだ」 | personal.json, episodes.json |

### 起動しないケース

- システム開発の指示・技術的な質問のみの場合
- 他人の話をしているだけで、自分のことに触れていない場合
- すでに同じセッション内で同じ情報を蓄積済みの場合

## 処理フロー

### 1. ユーザーの発言を分析

引数で渡されたテキスト、または直近の会話内容から以下を抽出する：

- **価値観・信念**: 「僕は〜だと思う」「〜が大事」「〜を信じてる」
- **経験・エピソード**: 「〜したことがある」「〜のとき〜だった」「前職で〜」
- **好み・習慣**: 「〜が好き」「毎日〜してる」「〜にハマってる」
- **スキル・専門知識**: 「〜ができる」「〜を使ってる」「〜年やってる」
- **振り返り・反省**: 「〜を学んだ」「〜は失敗だった」「今思うと〜」
- **目標・展望**: 「〜したい」「将来は〜」「次は〜に挑戦」
- **人間関係・コミュニケーション**: 「〜な人が好き」「〜タイプ」
- **キャラクター・口調の特徴**: 話し方の癖、よく使うフレーズ

### 2. 格納先を決定

抽出した情報を以下のファイルに振り分ける：

| 情報タイプ | 格納先 | 更新対象フィールド |
|-----------|--------|------------------|
| 口調・話し方の癖 | `profile.json` | `voice.vocabulary`, `voice.ending_patterns` |
| キャラ設定に関わる情報 | `profile.json` | `character.expertise`, `character.background` |
| 専門知識・技術スキル | `domain.json` | `expertise_areas[].key_facts`, `certifications` |
| 価値観・信念・人生哲学 | `personal.json` | `identity.values`, `identity.beliefs`, `identity.life_philosophy` |
| 仕事・キャリア経験 | `personal.json` | `experiences.career`, `experiences.achievements` |
| 失敗・学び | `personal.json` | `experiences.failures_and_lessons`, `experiences.turning_points` |
| 好み・趣味・日課 | `personal.json` | `preferences.likes`, `preferences.hobbies`, `preferences.daily_routines` |
| 嫌いなこと | `personal.json` | `preferences.dislikes` |
| エピソード・ストーリー | `personal.json` | `stories` |
| 振り返り・内省 | `personal.json` | `reflections` |
| 人間関係スタイル | `personal.json` | `relationships` |
| 具体的な実体験（投稿ネタ向き） | `episodes.json` | `episodes.{カテゴリ}` |
| 意見・スタンス・主張 | `opinions.json` | `opinions.{カテゴリ}` |
| ターゲット読者の新たな気づき | `target.json` | 該当フィールド |
| ジャンル・テーマの新方向 | `genre.json`, `strategy.json` | 該当フィールド |

### 3. 重複チェック

- 各格納先ファイルを読み込み、既存データと比較する
- **意味的に同じ情報**がすでに存在する場合はスキップする（完全一致だけでなく意味が同じものも）
- 既存情報を**補強・具体化**する内容の場合は、既存エントリを更新する
- 全く新しい情報のみ追加する

### 4. データ追加

対象ファイルを読み込み、適切なフィールドに情報を追加する。

追加時のフォーマット：

**配列にシンプルな文字列を追加する場合：**
```json
"既存の値と同じ形式の文字列"
```

**stories / reflections に追加する場合：**
```json
{
  "date": "2026-03-26",
  "content": "抽出した内容を簡潔にまとめたテキスト",
  "context": "どんな文脈で話されたか",
  "tags": ["関連タグ"]
}
```

**experiences 配下に追加する場合：**
```json
{
  "date": "2026-03-26",
  "content": "経験の内容",
  "lesson": "そこから得た学び（あれば）",
  "tags": ["関連タグ"]
}
```

### 5. 更新日時を記録

`personal.json` の `updated_at` を現在時刻に更新する。

### 6. ユーザーに報告

追加した内容を以下のフォーマットで簡潔に報告する：

```
📝 ナレッジ更新完了

追加:
- [格納先ファイル名] > [フィールド名]: 「追加内容の要約」

スキップ（重複）:
- 「既存と重複していた内容」

合計: X件追加 / Y件スキップ
```

## 実行手順

1. `data/pao-pao-cho/knowledge/` 配下の全ナレッジファイルを読み込む
2. ユーザーの発言内容を分析し、情報を抽出する
3. 重複チェックを行う
4. 新規情報のみ対象ファイルに書き込む
5. 結果を報告する

## ナレッジ深掘り質問機能

情報を蓄積した後、ナレッジの空白を見つけてユーザーに質問する。
これにより、投稿に「自分の経験や考え」を交えたリアリティと独自性を生み出す素材を集める。

### 質問生成の流れ

1. 全ナレッジファイルを横断的にスキャンする
2. 以下の観点で「まだ聞けていないこと」を特定する
3. 投稿の独自性に直結する質問を1〜3個生成する

### 質問の観点

| 観点 | 質問の狙い | 例 |
|------|-----------|-----|
| **原体験** | なぜその分野に興味を持ったか | 「AIに興味を持ったきっかけは？」 |
| **失敗談** | リアリティのあるエピソード | 「エンジニアとして一番やらかしたことは？」 |
| **こだわり** | 独自の視点・ポリシー | 「コードを書くとき絶対譲れないことは？」 |
| **日常習慣** | 共感を生む生活描写 | 「朝起きてまず何する？」 |
| **意外な一面** | ギャップで興味を引く | 「エンジニアっぽくない趣味ある？」 |
| **具体的な数字** | 説得力を出す素材 | 「AI導入で実際どれくらい効率上がった？」 |
| **価値観の変化** | ストーリー性のある投稿素材 | 「昔と今で一番考え方が変わったことは？」 |
| **テーマ別の実体験** | 各ジャンルの投稿に使える素材 | strategy.jsonのテーマごとに実体験があるか確認 |

### 質問の優先順位

1. **strategy.json のテーマに紐づく実体験が空** → 最優先（投稿に直結するため）
2. **personal.json の空配列** → 高優先（基本情報の欠落）
3. **domain.json にスキルはあるが具体エピソードがない** → 中優先
4. **既存情報をもっと深掘りできる** → 低優先

### 質問のトーン

- カジュアルに、会話の流れで自然に聞く
- 一度に大量に聞かない（最大3問）
- 「投稿ネタになりそうなんだけど」と目的を添えて聞く

### 報告フォーマット（質問パート）

ナレッジ更新の報告の後に続けて表示する：

```
💭 もっと知りたいこと

ナレッジを見て、投稿のリアリティを上げるために聞きたいことがあります：

1. [質問内容]
   → これが分かると「[テーマ名]」の投稿に実体験を交えられます

2. [質問内容]
   → [理由]

答えたくないものはスキップしてOKです！
```

## 注意事項

### 個人情報保護（重要）

以下の情報は **絶対に記録しない**：

- **本名・住所・電話番号・生年月日（年齢の大まかな世代はOK）**
- **勤務先の具体的な社名・プロジェクト名・顧客名**（`ng_words.json` の `company_ng` 参照）
- **第三者の個人を特定できる情報**（実名、所属、容姿の詳細など）
- **金融情報**（具体的な資産額、口座情報、給与の正確な額）
- **SNSアカウントのパスワード・トークン情報**

以下は **抽象化して記録する**：

| 発言の例 | 記録する形 |
|---------|----------|
| 「〇〇株式会社で働いてた」 | 「SaaS企業で勤務経験あり」 |
| 「年収は〇〇万」 | 「エンジニアとして平均以上の収入」 |
| 「〇〇さんと一緒にやった」 | 「同僚と共同でプロジェクト推進」 |
| 「〇〇区に住んでる」 | 記録しない |
| 「32歳」 | 「30代前半」 |

### その他

- ネガティブすぎる内容はそのまま記録せず、学びや気づきの形に変換する
- 曖昧な情報は「かもしれない」等の注釈をつけて記録する
